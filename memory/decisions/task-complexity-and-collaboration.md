## 核心原则

不依赖模糊的"复杂"/"简单"判断。用具体条件决定是否协作。

## 判定流程（5 秒决策）

```
收到任务 → 跑 5 个问题 → 多数 YES = 多 agent 协作 / 多数 NO = 单 agent 做
```

### 5 个判定问题

1. **任务能拆成 3+ 个互不依赖的子任务吗？**（不是步骤，是可以并行的独立模块）
2. **子任务涉及不同的文件集吗？**（不同目录、不同项目、不同语言）
3. **需要不同角色的专业能力吗？**（审核 vs 实现 vs 翻译 vs 调试）
4. **单 agent 做完需要超过 30 分钟吗？**（token 消耗大、上下文会爆）
5. **有明确的 spec/需求可以分发给其他 agent 吗？**（模糊需求不适合分发）

**≥3 个 YES** → 多 agent 协作
**≤2 个 YES** → 单 agent 做

## 具体场景速查表

### 单 agent 做（不需要协作）

- 单文件修改、bug fix（除非需要 cross-model review）
- 快速问答、解释代码
- 格式化、lint、简单重构
- 需求还模糊、还在探索阶段
- 紧耦合代码（改一处影响全局）
- 写一个函数、一个组件

### 多 agent 协作

- ARIS 科研流程（强制，不管复杂度）
- 多文件/多模块同时实现（3+ 独立模块）
- 需要审核的产出（代码 → Codex review，论文 → auto-review-loop）
- 大规模重构/迁移（今天做的 agentmemory 替换就是典型）
- 需要不同语言/领域知识（前端 + 后端 + 部署）
- 批量处理 + 质量检查（DeepSeek 批量 + Codex 抽检）

### 中间方案（不需要完整多 agent，但比纯单 agent 强）

| 方案 | 适用场景 |
|------|---------|
| 单 agent + 调用一次其他模型 | 写完代码后让 Codex 审一下 |
| Plan → Execute | 复杂 feature，先规划再实现 |
| 后台 agent | 明确的重构/迁移任务 |
| 循环 + 质量门 | 有测试可以验证的迭代任务 |

## 协作时的分工规则

| 角色 | 谁来做 | 什么时候调用 |
|------|--------|------------|
| 架构设计、全局决策 | CC (Opus) | 任务开始时 |
| 代码实现、部署 | OpenCode / CC | 设计确定后 |
| 审核、找漏洞、给方案 | Codex (GPT-5.5) | 实现完成后 |
| 翻译、摘要、批量文本 | DeepSeek | 有批量工作时 |
| 快速 lint、格式检查 | Haiku | 提交前 |
| 二次审查、测试建议 | Sonnet | 审核后 |

## 自主搜寻资料的规则

### 必须搜寻的情况

- 要用一个你不熟悉的库/框架/工具
- 要解决一个你没见过的错误
- 用户问了一个需要最新信息的问题（2025年之后的事）
- 没有现成 skill 且任务复杂度 ≥ 中等
- ARIS 流程中的 literature stress test 步骤

### 不需要搜寻的情况

- 已有 skill 覆盖（直接用 skill）
- 标准库/常见模式（你已经知道）
- 用户给了明确的实现方案
- 简单任务（单文件修改、格式化）

### 搜寻流程

1. 先查本地 `D:\agent-resources\` 有没有现成的
2. 没有 → 搜索 GitHub（热门、高质量、相关）
3. 找到有用的 → 下载到 `D:\agent-resources\repos\` 或 `skills\`
4. 提取关键信息用于当前任务
5. 如果是可复用的 skill/workflow → 安装到对应位置

### 软件开发时的反问规则

当需求模糊时（以下任一条件），agent 必须先反问用户再动手：
- 不确定用哪个技术栈/框架
- 不确定功能的边界（做到什么程度）
- 有多个合理的实现方案且 trade-off 不明显
- 涉及用户的业务逻辑（agent 不可能知道的）
- 会影响已有功能的改动

科研项目不需要反问（按 ARIS skill 执行即可）。
